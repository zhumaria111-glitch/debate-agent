## 一、Prompt Engineering 方法论

### 1.1 核心认知转变

从"会用 AI 聊天"到"以开发者视角调 prompt"——这是最大的认知跃迁。在和 chatbot 对话时，每次新问题要重新输入提示词，会下意识精进但不会特别重视。但以开发者视角来看，**prompt 是成本最低的调优手段**。在辩论分析项目中，3 轮迭代把综合分从 2.72 提到 4.24，绝大多数改进靠的不是改代码，而是改提示词。

更深一层的认知：**AI 产品的迭代不是越改越多代码，而是越改越准的 prompt。** 最难的那个 bug（辩论回合数据丢失）需要代码层修复，但影响最大的 bug（quick_view 空白）只需要两行 prompt 改动。这就是 prompt engineering 的杠杆效应——它不是"会说话就行"，而是投入产出比最高的调优手段。

- **Anthropic 官方 Prompt Engineering 文档**：[docs.anthropic.com](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering)
- **吴恩达 Claude Code 课程**：[DeepLearning.AI](https://learn.deeplearning.ai/courses/claude-code-a-highly-agentic-coding-assistant)

### 1.3 结构化 Prompt 技术

**1. 角色设定（System Prompt）**
System Prompt 是 prompt engineering 中最被低估但最强大的工具。它不是简单的"你是一个 XX 专家"，而是定义了整个对话的行为边界、输出格式、质量标准。

System Prompt 的四个核心要素：
- **身份**（Who）：你是什么角色，拥有什么能力
- **任务**（What）：你要完成什么
- **约束**（How）：什么不能做，什么必须做
- **输出格式**（Output）：返回什么格式的结果

**2. XML 标签结构化**
使用 XML 标签包裹不同类型的指令，是 Anthropic 官方推荐的结构化方法：
```
<instructions>核心任务描述</instructions>
<examples>少样本示例</examples>
<formatting>输出格式要求</formatting>
<constraints>行为和格式约束</constraints>
```

**3. 思维链（Chain of Thought）**
让模型在给出最终答案前，先展示推理过程。这能显著提高复杂任务的准确率。在辩论分析中，消融实验已验证 CoT 对准确率的增量贡献。

**4. 少样本学习（Few-Shot）**
给出 2-3 个高质量示例，比写一万字的说明更有效。AI 最擅长的是模仿——给它模板，它会比你描述得更准确地理解你的意图。

**5. Thinking Budget（思考预算）**
Claude 支持 extended thinking——允许模型在回答前进行深度思考。对于复杂分析任务（如辩论结构化），增加 thinking budget 可以显著提升输出质量。需要在速度和深度之间做权衡。

### 1.4 七条 Prompt 方法论（从个人项目中总结）

**场景一：自己很清楚要 AI 做什么（AI 扮演执行者）**

| # | 方法 | 具体操作 | 原理 |
|---|------|---------|------|
| 1 | **给示例和模板** | 给它一个模板，说清楚什么照抄、什么可改 | AI 比人更擅长模仿和复制——模仿比理解更快 |
| 2 | **负向约束优先** | 告诉 AI "不要做什么"的优先级远高于"要做什么" | LLM 对禁止指令的响应往往比正向指令更精确 |
| 3 | **细节越具体越好** | 头脑中所有具体细节都给 AI，它才能精准执行 | AI 不会"揣摩"你没说的需求，给越细越准 |
| 4 | **长期迭代思维** | 好 prompt 是改出来的。AI 做对的给正反馈，不对的明确修正 | 每次修改都是在训练 AI 理解你的标准 |
| 5 | **及时开新对话** | 对话跑偏时新开一个窗口，避免在错误基础上继续 | AI 的 context 会累积前面的错误，新窗口 = 清零 |
| 6 | **标签化表达** | 用 `关键词: 内容` 替代长段落 | 结构化输入让 AI 更快定位关键信息 |
| 7 | **控制输出格式、分步骤执行、让 AI 自我修正** | 设定具体身份 → 先确认模糊指令 → 分步完成 → 自行检查 | 把质量控制嵌入执行流程 |

**场景二：不太清楚自己要什么（AI 扮演咨询者）**

- **发散式提问**：告诉大致思路，让 AI 从几个角度发散
- **多角色咨询**：让 AI 同时扮演不同职业/身份的人给建议——现实中你需要问很多人，AI 可以一站式完成
- **苏格拉底式追问**：让 AI 不断追问你，帮你理清思路。这也是一种教育学的启发式方法——追问比给答案更有效

### 1.5 Prompt 工程的核心技术参数

**Temperature（温度）：**
- 越低（接近 0）→ 输出越确定、越稳定、越保守。适合需要精确一致输出的场景（如辩论结构化分析）
- 越高（接近 1）→ 输出越随机、越有创造性。适合需要多样性输出的场景（如头脑风暴、文案生成）
- 选择原则：需要准确性选低温度，需要创意选高温度

**Max Tokens（最大输出长度）：**
- 辩论分析项目中发现：max_tokens 设太小导致输出被截断，rounds 数据不完整
- v0.3 版本将 max_tokens 从 4096 翻倍到 8192，覆盖率显著提升
- 原则：宁可多给不要少给。输出不够时先检查 max_tokens 是否设小了

**Top-P / Top-K：**
- 控制输出多样性的另外两个参数
- 一般配合 Temperature 使用，日常场景中 Temperature 调参优先级更高

### 1.6 System Prompt vs User Message 的分工

| | System Prompt | User Message |
|------|-------------|-------------|
| **作用** | 定义"你是谁"、"你该怎么做" | 表达"这次要做什么" |
| **改变频率** | 低，产品级固定的 | 高，每次对话不同 |
| **内容** | 角色、规则、格式、边界 | 具体任务、输入数据、上下文 |
| **示例** | "你是辩论分析专家。只输出 JSON。" | 辩论文字稿全文 + "请分析" |

**核心原则**：System Prompt 写规则，User Message 写数据。不要把规则和任务混在一起。

### 1.7 让 AI 先讨论方案再执行

一个重要的 prompt 技巧：让 AI 在真正处理前先讨论方案，把思路聊清楚再具体执行。这类似于 Plan 模式——先对齐认知，再动手。对于复杂任务，这个额外的讨论步骤能大幅降低返工率。

### 1.8 让 AI 自查与互查

- **自查**：AI 生成方案后，让它转换身份（如"你是评委"）反过来评估自己生成的结果。注意 context 要保持干净，建议新开对话
- **互查**：换一个模型来评估另一模型生成的输出
- **强制 AI 承认错误**：直接让模型审查 bug 它会说一切正常，但告诉它"这里有一个 bug"，它会反复循环并最终发现问题。Josh Pigford 的 "Butt For Real" Skill 和 OpenClaw 创始人 Peter Steinberger 都验证了这一点

### 1.9 标签化表达与结构化输入

相比于输入一段长文让 AI 从中提取关键信息，直接用"关键词: 内容"的标签化格式输入，能让 AI 一目了然它需要做什么。这是人机交互中一个被低估的技巧——**输入的结构化程度 = 输出的质量上限。**

### 1.10 AI 输出质量归因的四变量模型

当 AI 生成效果不好时，不是盲目怀疑"AI 不行"，而是系统性地从四个维度归因：

| 变量 | 排查方法 | 典型问题 |
|------|---------|---------|
| **代码** | 检查是否有逻辑错误 | 数据未正确传入、结果未正确合并 |
| **模型** | 换模型对比 | 同一 prompt 在不同模型下表现差异大 |
| **提示词** | 加/改约束，对比结果 | 缺少负向约束、输出格式要求不明确 |
| **预处理** | 控制其他变量，对比不同预处理数据 | 数据碎片化、段落边界破坏结构 |

**核心方法**：控制变量法。保持其他条件不变，只改变一个因素，对比结果差异。这就是产品经理的"A/B Test 思维"用在 prompt debugging 上。

### 2.2 Claude Code 深度使用

Claude Code 是 Anthropic 官方出品的 **Agentic Coding Assistant**——它不是一个代码补全工具，而是一个能自主探索代码库、执行多步骤任务、使用工具、记住上下文的编程 Agent。

#### 2.2.1 安装与环境配置

- 在 VS Code 中安装 Claude Code 扩展
- 安装 **CC Switch** 插件，可一键切换模型
- 购买 API 中转服务（≈20 美元），可使用 Claude Code 全部模型
- 配置国内模型：DeepSeek、MiniMax、GLM、Qwen、豆包均可通过 CC Switch 一键切换

#### 2.2.2 CLAUDE.md：Agent 的项目记忆系统

CLAUDE.md 是 Claude Code 的核心机制——它让 Agent 在每次对话中"记住"项目上下文。

| 文件 | 位置 | 创建方式 | 可见范围 | 用途 |
|------|------|---------|---------|------|
| `CLAUDE.md` | 项目根目录 | `/init` 自动生成 | 团队共享 | 项目运行方式、测试规范、代码规范、开发行为约束 |
| `CLAUDE.local.md` | 项目根目录 | 手动创建 | 个人（加到 `.gitignore`） | 个人偏好配置、本地开发环境差异 |
| `~/.claude/CLAUDE.md` | 用户目录下的 `.claude` 文件夹 | 手动创建 | 本机所有项目 | 全局生效的规则和偏好 |

**CLAUDE.md 标准结构（四段式）：**
1. **项目基础介绍**：项目简介、技术栈、核心依赖
2. **项目目录结构与文件分工**：每个目录/关键文件的作用
3. **运行规则、部署规则、测试规则**：`npm run dev`、`npm test`、构建命令、环境变量
4. **开发行为约束**：代码风格、提交规范、禁止事项

**关键认知**：CLAUDE.md 写得越好，Claude Code 的表现越稳定。它不是一次性写完的——每次发现 Agent 反复犯同一个错误时，就把规则写入 CLAUDE.md。

#### 2.2.3 Slash Command 体系

| 命令 | 功能 | 使用场景 |
|------|------|---------|
| `/init` | 自动分析项目并生成 CLAUDE.md | 新项目第一次使用 |
| `/help` | 列出所有可用命令及简要说明 | 忘记命令时 |
| `/clear` | 清空当前对话历史 | 对话跑偏、切换任务 |
| `/compact` | 压缩上下文，释放 token 空间 | 对话过长、token 即将耗尽 |
| `/escape` | 中止当前正在执行的任务 | Agent 跑偏或执行错误操作 |

**快捷操作：**
- `#` 开头写文字 → 注释/指令
- `@文件名` → 引用特定文件让 Agent 关注
- `Tab` → 补全代码或命令

#### 2.2.4 三种执行模式

重大修改前**必须**先进入 Plan 模式。在 VS Code 中内置三种模式：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **Plan 模式** | Agent 先输出计划，你确认后再执行 | 架构变更、跨文件重构 |
| **每次询问模式** | 每步操作前征求你的同意 | 敏感文件改动 |
| **直接执行模式** | Agent 直接修改代码，无需确认 | 明确的小改动 |

**Pro tip**：在终端中使用 `Shift+Tab` 两次打开 Plan 模式。重大修改如果不先在 Plan 模式下讨论清楚方案，Agent 容易"鲁莽自信"地直接执行。

#### 2.2.5 MCP（Model Context Protocol）深度理解

MCP 是 Anthropic 推出的**模型上下文协议**——让 LLM 能够调用外部工具和数据源的标准化协议。

**一句话理解：** API 是程序之间的通信方式，MCP 是 LLM 和外部工具之间的通信方式。

**MCP Server 配置：**
```bash
claude mcp add <server-name>
# 例如：添加 Playwright 浏览器自动化工具
claude mcp add playwright
```

**常见 MCP Server 与应用场景：**
- **Playwright**：浏览器自动化，截图、分析页面、端到端测试
- **Figma MCP**：直接读取 Figma 设计稿，自动生成前端代码
- **Filesystem**：安全的文件系统访问
- **GitHub**：仓库操作、PR 管理
- **自建 MCP**：可以连接飞书等通讯工具

**关键认知**：MCP 让 Claude Code 从"代码编辑器"变成了"全能开发环境"——接上 Figma 可以直接把设计稿转成代码，接上飞书可以通过聊天触发开发任务。

#### 2.2.6 Agentic Workflow 核心模式

Claude Code 的工作流程体现了 Agent 设计的核心范式：

1. **探索阶段**：Agent 先阅读项目结构、CLAUDE.md、关键文件，建立上下文
2. **规划阶段**：拆解任务为子步骤，评估每个步骤的依赖和风险
3. **执行阶段**：逐步骤执行，每一步完成后自我校验
4. **验证阶段**：运行测试、检查 lint、确认改动符合预期
5. **记忆阶段**：将本次学到的项目规律写入 CLAUDE.md

**一个典型的坑**：Claude Code 有时"很鲁莽且自信"——你让它带着你做，它二话不说全做完了。应该明确告诉它"做事情前要告诉我原因（通俗版解释），不要轻易直接行动"。或者直接使用 Plan 模式。

#### 2.2.7 Hook 系统

Claude Code 支持 Hook 机制——在特定事件触发时自动执行自定义脚本或行为。这是从"使用工具"到"定制工具"的关键跃迁。通过配置 `settings.json` 中的 hooks，可以实现：
- 工具调用前后自动执行检查
- 会话开始/结束时的自定义行为
- 特定文件改动后的自动化流程

### 2.3 OpenClaw 深度使用

OpenClaw 是一个**可部署的个人 AI Agent 框架**——它不是对话框，而是一个完整的数字实体，能 7×24 小时在后台运行。

#### 2.3.1 核心架构：四大能力模块

| 模块 | 功能 | 技术实现 |
|------|------|---------|
| **Tools** | 接入外部能力与 API | MCP Server、自定义 API 集成、文件系统访问 |
| **Memory** | 存储历史对话与用户偏好 | 双层记忆：短期（上下文窗口）+ 长期（向量检索） |
| **Planning** | 拆解复杂目标，制定执行步骤 | 任务分解 → 依赖分析 → 分步执行 |
| **Actions** | 在真实环境中执行具体操作 | 终端命令、API 调用、文件操作、消息发送 |

**核心理念**：超越自动化，告别僵化流程，走向**动态决策**。

#### 2.3.2 部署与配置实战

**部署路径（我的实际经历）：**
1. 在终端中安装部署 OpenClaw
2. 创建飞书机器人并连接 OpenClaw（三个机器人分别用于不同场景）
3. 配置 Skill：让 Agent 获得特定领域能力
4. 连接通讯软件：飞书、微信等，实现手机上随时与 Agent 交互
5. 探索 Subagent：创建多机器人协作体系

**关键踩坑：**
- 在对话框中让 Agent 创建新机器人时，它把原来的主机器人配置搞没了——原因是对话框方式虽然方便但缺少精确控制
- **教训**：能用 CLI 终端操作的就不要在对话框操作。每次对话有上下文记忆，累积的 token 消耗量远超预期

#### 2.3.3 CLI 优先原则

**为什么 CLI 比对话框更高效：**

| 操作方式 | Token 消耗 | 精确度 | 适用场景 |
|---------|-----------|--------|---------|
| 对话框 | 高（每次都带上下文） | 中 | 探索性任务、聊天式交互 |
| CLI 终端 | 低（无上下文开销） | 高 | 管理、查看、配置、部署 |

**必须养成的 CLI 习惯：**
- 查看状态、列出 skill、管理 subagent → 用 CLI
- 复杂问题讨论、需求分析 → 用对话框
- 配置修改、部署上线 → 先用 Plan 模式讨论，再用 CLI 执行

#### 2.3.4 Subagent 系统

OpenClaw 支持创建多个 Subagent，每个 Agent 有独立的：
- 系统提示词（定义角色和能力）
- 工具集（不同的 MCP Server 和 Skill）
- 记忆空间（独立的上下文和偏好）

**多 Agent 协作实践：**
- 数据分析场景最适合多 Agent 并发——调用多个 Subagent 同时处理不同维度的数据
- 目标驱动的自主执行：给一个目标，让 Agent 自己拆解、分配、协调多个 Subagent
- 创建 AI 产品分析机器人、学习助手机器人等不同角色的 Agent

**踩坑记录**：创建 Subagent 时要清楚配置的继承关系，否则主 Agent 配置可能被覆盖。

#### 2.3.5 Skill 开发与安全

**Skill 本质**：可复用的能力模块，定义了特定任务的输入/输出/边界。

**安全原则**：在用别人的 Skill 之前，必须先看懂 Skill 代码——防止里面有恶意行为被 Agent 直接执行。这是 AI Agent 时代的新安全范式：不是"不要装不明软件"，而是"不要运行看不懂的 Skill"。

**Skill 开发方法论：**
1. 先用对话框让 Agent 完成一次任务，观察它的工作流程
2. 导出聊天记录，让 AI 分析流程，提炼为可复用的步骤
3. 把步骤写成 Skill，加入错误处理和边界判断
4. 反复测试和修改，直到稳定运行
5. 沉淀到 Skill 库，后续类似任务直接调用

#### 2.3.6 Cron 定时任务

OpenClaw 的 Cron 功能可以让 Agent 定期自动执行任务——定时提醒、每日资讯推送、定期数据检查等。这是"AI 主动服务你"的开端。

#### 2.3.7 Token 优化策略

通过 CLI 降低 token 消耗的核心方法：
- 能用终端命令的不开对话框
- 给 Agent 精确的指令，减少"来回确认"的 token 浪费
- 限制上下文长度，定期 `/compact`
- 理解 Agent 的原理（上下文记忆机制）才能从根源上优化

### 2.5 Skill、Rule、Memory 体系

这是现代 AI Agent 的三个核心概念，跨越 Claude Code、OpenClaw、Cursor 等所有主流 Agent 框架。

| 概念 | 定义 | 技术实现 | 示例 |
|------|------|---------|------|
| **Skill** | 可复用的能力模块 | Markdown 文件，定义输入/输出/边界/步骤 | AI 信息源整理 Skill、Learnings 复盘 Skill |
| **Rule** | 行为约束规则 | 配置文件中的规则语句 | "修改前先告知原因"、"不要虚构数据" |
| **Memory** | 跨会话持久记忆 | 双层架构（工作记忆 + 情景记忆） | 项目偏好、用户习惯、历史决策 |

**关键实践循环：**
```
让 AI 分析你的工作流 → 生成 Skill → 沉淀为可复用模块
  → 每完成一个阶段运行 "Learnings" Skill
    → AI 回顾对话，总结反复修改才改对的点
      → 写入 CLAUDE.md（Rule）→ 下次对话自动生效
```

**这个飞轮的价值**：你使用 AI 的方式本身在迭代优化——不是你在适应 AI，而是 AI 在适应你。

1. **Prompt Engineering**（提示词工程）→ 关注"怎么跟 AI 说"
2. **Context Engineering**（上下文工程）→ 关注"给 AI 什么信息"

## 四、模型技术基础

### 4.1 LLM 预训练全流程

**Deep Dive into LLMs 学习笔记：**

1. **预训练（Pretraining）**：为模型收集足够多的高质量语料，使其成为拥有庞大知识库的基础模型
   - 数据来源：Common Crawl（公开大规模网页抓取数据集）
   - URL 过滤 → 文本抽取 → 语言过滤 → PII 过滤（隐私信息）
   
2. **Tokenization（分词）**：
   - Unicode：将不同语言文字和符号用编号表示
   - UTF-8：将编号转换为二进制（目前全球互联网最通用字符编码）
   - 压缩原理：将 8 个 bit 的序列组固定为一个数字（0-255），大幅压缩编码长度
   
3. **神经网络原理**：期待一维的符号序列，需要有限的可能符号集合

### 4.2 端到端 AI 问答技术环节

用户提问 → 分词（切成数字编号）→ 服务器三道安检（鉴权/限流/路由）→ GPU 反复猜下一个词（推理）→ 流式传输（每猜出一个词立刻推送）→ 后处理（格式和安全检查）→ 前端渲染

## 五、模型评估体系

### 5.1 学习路径

```
Google MLCC 模型评估章节 → Hugging Face Evaluate（看指标解释）
  → Andrew Ng 评估相关章节 → Kaggle 比赛评估方式
    → OpenAI Evals 文档 → DeepLearning.AI LLM Evaluation
      → 读一个大模型评估报告 → 自己做一个小型评测集
```

### 5.2 分类任务指标

**混淆矩阵（以垃圾邮件识别为例）：**
- **TP（真正例）**：预测为垃圾，实际是垃圾 ✓
- **TN（真负例）**：预测为正常，实际是正常 ✓
- **FP（假正例）**：预测为垃圾，实际是正常——误杀，用户收不到正常邮件
- **FN（假负例）**：预测为正常，实际是垃圾——漏网，可能造成安全问题

**核心指标：**
- **精确率（Precision）**：TP/(TP+FP) —— 模型预测为真的里面，有多少是真的。重点看 FP，衡量"误杀"
- **召回率（Recall）**：TP/(TP+FN) —— 所有真的里面，模型找出了多少。重点看 FN，衡量"漏网"
- **准确率（Accuracy）**：(TP+TN)/全部 —— 所有预测正确的比例
- **F1 Score**：精确率和召回率的调和平均，惩罚"偏科"的模型
- **AUC**：模型本身的区分能力。高 AUC 的模型，后续调阈值更容易兼顾误杀和漏判；低 AUC 的模型怎么调都难两全

**面试高分回答模板：** "评估模型时，我会先看 AUC 判断模型整体区分能力，优先选择 AUC 更高的模型；再根据业务场景，看精确率、召回率和 F1，选择最适合业务的阈值和最终方案。"

**阈值选择：**
- 阈值高（如 0.68）：宁可错过，不可误杀 → 精确率高，召回率低
- 阈值低（如 0.25）：宁可错杀，不可错过 → 精确率低，召回率高

**三种数据分布类型：**
- 正常型：正负分布均衡，可通过调整阈值平衡精确率和召回率
- 重叠型：正负分布重合，无论怎么调阈值结果都差
- 极度不平衡：负例远多于正例。如果模型全说正常，还有 99% 准确率，但实际一个垃圾邮件都没识别

### 5.3 回归任务指标

- **MAE（平均绝对误差）**：各误差同等重要时选用
- **MSE（均方误差）**：大误差比小误差严重得多时选用
- **RMSE（均方根误差）**：对大偏差敏感的场景

### 5.4 生成式 AI 评估指标

**NLP 自动指标：**
- **BLEU**：评估翻译质量，看机器译文与参考译文的重合度。局限：只看词语重合，不考虑语义
- **ROUGE**：评估摘要质量，关注生成文本对参考答案的覆盖率（召回视角）
- **困惑度（Perplexity）**：语言模型常用指标，衡量对生成序列的预测能力。困惑度越低，模型越"自信"

**人工评估维度（四个维度打分）：**
- 相关性：答案是否切题
- 流畅度：语言是否自然
- 事实性：内容是否准确
- 有用性：是否真正解决用户问题

**幻觉率：** 模型生成虚假但看起来合理的内容比例。需要单独建立评估体系：抽样人工核查 + 基于知识库的自动核查

### 5.5 过拟合与欠拟合

- **过拟合**：训练集表现极好但新数据表现差。像只会死记硬背的学生，不会灵活变通。解决：收集更多样化数据、设计更严格的离线评测集
- **欠拟合**：训练集和测试集表现都差。像什么都没认真学的学生。解决：评估是否需要更强的基础模型、补充关键训练数据

### 5.6 A/B 测试与统计基础

**核心概念：**
- **置信区间**：对真实值所在范围的估计。95% 置信区间 = 重复 100 次实验，95 次真实值落在区间内
- **P 值**：实验结果是纯属巧合的概率。< 0.05 说明改动有效，> 0.05 更可能是随机抖动
- **α（显著性水平）**：允许出错的空间，通常 5%
- **1-β（统计功效）**：把真正有效的改动抓出来的概率，通常 80%
- **MDE（最小可检测效应）**：预期改动能涨多少。越小需要的样本越多

**常见 A/B 测试的坑：**
1. **辛普森悖论**：两组用户构成不同影响结果 → 确保实验组和对照组用户特征分布一致
2. **新奇效应**：刚上线的改动因好奇心导致指标虚高 → 延长观察周期
3. **网络效应干扰**：社交/协作类产品中 AB 组互动污染结果 → 按独立集群分流（如地域）
4. **多重检验问题**：仅凭运气 → 提前声明主指标，对辅助指标做校正

### 6.3 鹦鹉巧辩（黑客松项目）

**获奖：** 2026 年 5 月抖音 AI 创变者计划黑客松联赛山东赛区三等奖

**产品定位：** 面向全年龄段网民的批判性思维训练互动平台，以辩论赛逻辑为核心，通过轻量化、趣味化的竞技玩法解决思维固化、信息茧房问题

**核心玩法：**
1. **田忌赛马式卡牌策略对战**：AI 分析用户选取的视频生成 5 张论证卡牌（双方总体论证强度一致，但每张参差不齐），用户排序卡牌论证力 → 公开式三轮出牌对战
2. **互换身份复活赛**：双方互换立场，从对方卡牌中挑出最有论证力的观点，补充论据完善逻辑，由原立场用户评判

**产品价值：** 从自主分析论证 → 策略化对战 → 沉浸式换位思考，完成"判断-思考-转变"的思维蜕变

**48 小时黑客松收获：**
- 完整流程体验：idea → vibecoding demo → 游园会推销 → 收集反馈 → 迭代

**好产品 idea 的来源：** 平时多体验产品、游戏、电影、书籍，兴趣驱动的力量远大于被动学习

### 7.1 AI 产品经理核心能力模型（来自美团、腾讯面试官反馈）

- **要有编程视角**：理解模型基础运行逻辑、输入输出方式，才能正确定义问题
- **懂代码逻辑和技术边界**：不需要自己写，但能理解技术边界
- **AI Native 思考**：从"能不能用 AI 解决"出发重新审视所有业务流程
- **用户洞察和品味更加重要**：AI 降低了执行门槛，产品经理的价值向需求理解和审美倾斜
- **动手最重要**：亲手做一个小项目比看一百个攻略都有用。项目不一定大，落地零到一
- **持续学习**：对 AI 保持足够的热情和好奇心，深度使用了解 AI 能力边界

### 7.2 AI PM 与传统 PM 的差异

- 传统 PM：先写 PRD → 提需 → 等研发排期 → 周期很长
- AI PM：自己手搓 demo 验证想法 → 把原型当 PRD → 研发介入做工程化
- 花更多时间在与模型对话上，而非画原型图
- 重复性工作没必要做了，团队内部都在做 skill
- 岗位边界更模糊，能力要求更复合

### 7.3 用户研究方法论

**核心原则：观察行为，而非倾听需求。** 用户说的需求往往是解决方案，真正的需求藏在行为和挫败感里。

**三方法组合：**
| 方法 | 目的 | 规模 | 周期 |
|------|------|------|------|
| 实地观察 | 看真实行为，发现用户说不出的痛点 | 5-8 所学校 | 1 周 |
| 深度访谈 | 挖掘动机、挫败感、现有工具使用习惯 | 20-30 人 | 2 周 |
| 定量问卷 | 验证假设、量化需求优先级 | 500 人以上 | 1 周 |

**深访问题设计（行为还原式提问）：**
- "上周你备了哪节课，能跟我描述一下整个过程吗？"（还原真实行为）
- "哪个环节让你最头疼或最费时间？"（挖痛点）
- "你现在用什么方式解决这个问题？"（了解替代方案）
- "如果有个工具能帮你做 X，你会担心什么？"（挖阻力）

**调研输出三产物：** 用户画像 → 用户旅程地图 → 需求优先级矩阵（按频次 × 痛苦程度排序），直接对接 PRD

### 7.4 Demo 到产品的 Gap 分析

以保险工单分类项目为例，系统梳理 12 个差距维度：

| 维度 | Demo 状态 | 生产环境要求 |
|------|----------|------------|
| **数据** | 静态 TSV 文件 | 实时数据流（CRM API/消息队列/数据库） |
| **标注** | 无标注流程 | 标注工作流 + 标准文档 + 仲裁机制 + 标签版本管理 |
| **多意图** | 单标签 | 一条工单可能属于多个类别 |
| **标签体系** | 固定 | 粒度/互斥性/业务对齐/兜底策略 |
| **延迟与成本** | 不计 | 单次耗时 + token 消耗实时监控 |
| **置信度** | 无 | 阈值 + 人工审核 |
| **向量数据库** | 随便选 | 对比 Pinecone/Weaviate/Milvus/pgvector 的延迟/成本/运维 |
| **评估监控** | 一次性 | 持续评估 + 数据漂移检测 + A/B Test + 业务指标挂钩 |
| **异常降级** | 无 | 每条路径都要有兜底流程图 |
| **性能** | 本地 | 并发压测 + 延迟优化 + 高峰期稳定 |
| **交互** | 命令行 | API 封装 + 客服系统对接 + 前端审核看板 |
| **迭代闭环** | 一次性 | 人工审核 → 错误样本回采 → 微调 → 提示词优化 |

### 7.5 消融实验方法论

四个策略递进验证：
- **策略 0：Random Baseline** — 最低门槛对照
- **策略 1：简单 Prompt** — 证明 LLM 有基础能力
- **策略 2：+RAG** — 证明 RAG 对准确率的边际贡献
- **策略 3：+CoT** — 证明思维链的增量效果

关键：不仅要看准确率，还要综合考虑精确率、召回率、F1，以及与成本的关系。用数据对接技术团队改进，用业务指标对接业务方说明影响。

### 7.6 AI 替代人类工作的最佳场景

**必要性工作 + 机械重复 + 低价值消耗 = AI 出场。** 例如 Abridge 的语音转病历——写病历是必要的合规工作，但医生不应该把时间花在文书上，应该花在患者沟通上。

### 7.7 腾讯/美团 AI PM 面试思考题（部分）

- AI 项目 prompt 优化、模型评测方法
- 对话流中 AI 什么时候应触发 RAG，什么时候应直接回复——模型幻觉理解 + 检索成本与生成质量的权衡
- 如何验证准确率提升来自 RAG 优化而非模型随机波动（A/B Test、Baseline 设定、统计显著性）
- 为何针对法律/医疗/编程定制不同 Agent 插件——ROI 题目：差异化效果增益是否覆盖研发维护成本
- AI 功能做成独立 App 还是嵌入微信/QQ 的功能插件（专而精 vs 大而全）
- 模型推理成本过高与用户追求极致长文本输入冲突时的处理方案
- 假如周留存率下滑 10%，如何系统性拆解原因（模型表现、响应速度、竞争环境、用户侧变化）
- 和技术团队对"模型效果是否可上线"产生冲突时的解决方式
- 如果 AI 可以自动写代码画图设计原型，PM 的核心价值是什么

## 八、行业播客深度笔记

### 8.1 Abridge：AI 语音转病历

**痛点：** 医生在诊室外花费大量时间写病历——必要性工作 + 机械重复 + 低价值消耗

**核心技术壁垒（四层）：**
1. 音频采集与预处理（降噪、去混响）
2. 语音转写（专业模型 + 微调 + 实时流处理）
3. 临床语义理解——实体识别（20+ 类临床实体：主诉、症状、既往史、过敏史、检查结果、医嘱等）+ 关系抽取（实体之间因果关系）+ 上下文消歧（模糊表述结合上下文定位）
4. 结构化病历生成

**强制可追溯机制控制幻觉：** 模型每生成一句话，都要严格对应原始对话中的音频或文字片段，医生可随时点击验证。这个机制可能在一定程度上解决医患纠纷。

**AI 评估体系：** 用"对抗样本"测试极端场景 + 与顶尖医学机构合作 + 严格的红线机制

### 8.2 Anthropic PM：如何打造下一代 Claude

**Alex Albert（Anthropic Research Team PM）的核心观点：**

- **AI PM 关注的是 Gap**："我们想让它擅长的能力"和"它本身擅长的能力"之间的差距。因为 AI 有不可预测性，产品规划不能只靠需求列表
- **Next Claude 的核心方向**：重点关注上一代模型没解决的问题，通过用户 feedback 驱动
- **Context 决定一切**：用户抱怨回答"过快过于简单"，根因是没有给模型足够 context。把自己信息做成文件喂给模型，质量天差地别
- **AI 对 PM 的改变**："not blocked as much on getting to answers and getting to data"——获取知识更快，打破岗位之间壁垒
- **Dreaming**：Claude 也会像人一样睡觉时自动整理和巩固记忆
- **Try it out**：自己做一遍然后和 Claude 做比对，这是理解模型能力边界最快的方式
- **Anthropic 对文档的重视**：很多会议从阅读文档开始——对 Claude 来说，文档本身就是高质量训练数据
- **Agent 的上下文管理**：什么时侯需要人工介入，哪些地方 Agent 卡住了——这是 PM 要持续思考的问题
- **摆数据说话**：告诉团队"目前有百分之多少的用户遇到这个问题"

### 8.3 MCP 的深度解析（Stainless × Anthropic）

**Alex Ratray（Stainless 创始人兼 CEO，公司被 Anthropic 以 3 亿美元收购）：**

- **API vs MCP**：API 是程序之间的沟通方式，MCP 是 LLM 和外部工具之间的沟通方式。在 Agent 兴起之前是人通过 UI 操作电脑，Agent 兴起后通过 MCP 和计算机交互
- **MCP 落地三大难点**：工具多且接口多（上下文限制 + 冗余信息干扰）、安全与权限问题未标准化、工具名称和描述必须极其精准
- **好 MCP Server 四原则**：深入市场观察真实场景 → 搭建完善评估体系搭配不同客户端/模型 → 工具越少质量越好 → 名称和描述必须精准
- **三种 MCP 方案**：全部封装（适合小 API）、精选封装（精简工具数）、动态模式（只开放 3 个核心工具：查询/了解/执行，完美解决上下文限制但每次需 3 次交互）

**MCP 检索的"超级工具"方案：**
- 与其让模型调用海量独立工具，不如给它两个超级工具：代码执行工具 + 文档检索工具
- 模型直接写 TypeScript 代码调用 API SDK，代码中完成所有操作
- 几乎不占用上下文窗口，仅需少量 token
- 未来代码执行工具会成为最主流的工具
- **当前局限**：模型调用第三方库能力差——无法精准匹配版本、容易虚构接口、调试困难

**未来预测——人机共生体（Cyborg）：**
- 一半是 LLM 神经网络，一半是传统 CPU 代码
- 两大场景：一次性业务操作 + 商业化生产软件
- 高频有用的 AI 操作会逐渐固化为标准化软件功能：手动 → AI 自动 → 固化为标准化功能
- 对话界面适合探索，仪表盘适合高频需求
- 有了代码执行超级工具后，"工具开发"本质就是提示词工程

**安全模型**：安全管控应在 API 层面而非限制 MCP 功能。基于 OAuth 协议实现精细化权限管控。

### 8.4 Gemini 核心团队：AI 的五个前沿方向

**Oriol Vinyals（Gemini 技术负责人）：**

1. **视觉 AI 的"GPT 时刻"尚未到来**：脱离文字、从纯视觉数据中提取抽象概念是关键突破口。表征学习仍停留实验室阶段，落地将颠覆行业
2. **Google Omni 世界模型**：可操控的现实渲染器，通过语言指令操控静态图片、模拟真实物理行为。深度适配自动驾驶与机器人——机器人实景数据优化模型，虚拟仿真场景低成本批量生成训练数据
3. **Gemini Spark 智能体**：从专用到通用。通用多域模型 + 底层架构 > 单场景定制。未来人工搭建的多 Agent 协作框架将被淘汰，模型自主生成适配任务的架构。推理模型将自主判断推理难度与思考时长
4. **AI 双层记忆**：工作记忆（上下文窗口）+ 情景记忆（检索历史）。短期最优落地方案：公私双知识库——私有库保障个性化与隐私，公共库沉淀通用能力
5. **RL 后训练蓝海**：拥有无限数据潜力，存在可验证性瓶颈。初创团队不应自研底层模型，应深耕垂直场景、搭建专属行业知识库——成本更低、落地更快

### 8.5 Josh Pigford：AI 时代的创业法则

1. **ADHD 成了优势**：AI 弥补执行力短板，同时做 5 个以上产品，"兴趣广"和"想得快"反而成了生产力
2. **Butt For Real Skill**：强迫 AI 承认它有地方搞砸了。这一点与 OpenClaw 创始人 Peter Steinberger 的发现一致——直接让模型审查 bug 它说一切正常，告诉它"这里有一个 bug"它就能发现
3. **AI 解决具体的麻烦**：为患病家人开发医疗文档对话工具。在 AI 时代对人的需求的探索应该更微观
4. **快速上线**：做了 25 年产品、卖过 400 万美金的公司，依然说"每次上线还是会恐惧"。解药是"快"——24 小时上线原型，撞南墙比写 PPT 强
5. **让 AI 主动复盘**：每完成阶段运行 Learnings skill，把反复改才改对的地方写进 CLAUDE.md

### 8.6 Anthropic 用户行为报告：AI 辅助决策四大赛道

6% 的用户会向 Claude 寻求个人指导。超过四分之三集中在四个领域：

| 赛道 | 占比 | 核心矛盾 | 细分方向 |
|------|------|---------|---------|
| 健康与养生 | 27% | 决策信息不对称，后果自己扛 | 私人医生、营养搭配、康复管理、用药助手 |
| 职业与事业 | 26% | 跳槽/转型风险高，缺乏客观参谋 | 求职策略、跳槽评估、职业转型、副业方向 |
| 人际关系 | 12% | 社交没有标准答案，需要参谋 | 向上社交、说服沟通、亲密关系、家庭调解 |
| 个人理财 | 11% | 金融知识门槛高，试错成本大 | 消费建议、保险匹配、资产配置、大额支出 |

**AI 决策辅助产品的四个特质：** 专业（调取靠谱信息，不是拍脑袋）、易懂（说人话，像聪明的朋友）、中立（不谄媚，敢于指出你的错误）、隐私保密（用户把 AI 当最信任的人，这份信任不能辜负）

**产品边界：AI 负责信息处理和推演，人保留最终选择权。**

## 九、学习方法论

### 9.1 五步学习循环

```
听播客（建立认知框架，把自己代入情境）
  → 动手改项目（在实践中发现问题、验证认知）
    → 随时记录心得（碎片化输入系统化输出）
      → 晚上全盘复盘（把散点串成线）
        → 根据复盘补充下一轮学习内容
```

**核心信条：学习和实践相辅相成。** 先听一个播客，总结输出，接着动手改项目，改的过程中随时记录心得，晚上做全盘复盘反思。在项目中学习和单纯学理论有本质区别——项目给了方向感和验证场。

### 9.2 Context 是个人最重要的数字资产

"学习是先把书读厚，慢慢再变薄"。积累 context 也是一样——刚开始把与 AI 的全部对话导出来，慢慢让 AI 解释每一步操作。长此以往：
- 你会越来越清楚 AI 在做什么、为什么这么做
- 剩下的东西都是自己真正理解的
- 可以根据自己的理解整理 AI 能看懂的 context
- Context 积累和个人知识库相辅相成——很多人说知识库已落后，但对个人来说它仍然重要

**终极目标：把自己蒸馏给 AI。** AI 越熟悉你，就越能成为得力助手。

### 9.3 从名词解释开始

学习全新领域的第一步是搞懂基本概念。什么时候 AI 能实现"光标放到某个词上自动解释"——这是一个产品 idea。在 AI 实现之前，自己用 AI 工具主动查概念，建立术语体系。

### 9.4 讲解是最好的学习

准备录制 AI 工具教程视频的过程本身就是学习闭环：
- 必须搞清楚工具的输入/输出/边界才能讲清楚
- 讲完会发现理解又深了一层
- 学习、思考、实践形成良性循环——"learn in public"

### 9.5 AI 可替代哪些工作

思考上一段实习（京东采销）中哪些是重复性机械劳动——比价、上下柜、平行仓开通、手工单处理——这些都是 AI 可以替代的。每一个"我当初如果有这个工具就好了"的念头，都可能是一个产品 idea。

### 9.6 保持好奇心与动手习惯

- 每天拆解一个 AI 应用：看它做什么、怎么赚钱、为什么好/不好
- 看到问题先想"能不能用 AI 解决"
- 平时多体验产品、游戏、电影、书籍——兴趣驱动的力量远大于被动学习
- "亲手做一个小项目比看一百个攻略都有用"——美团 AI PM 面试官的原话
