### v0.4.2 关键改进：Round 结构

**v0.4.1 输出的 rounds（错误）：**

```
立论（正方）→ 质询（反方）→ 自由辩论 → 结辩（反方）→ 结辩（正方）
```

强行将哲理辩套入标准竞技辩论模板。「自由辩论」和「质询」这两个环节在这场辩论中根本不存在。

**v0.4.2 输出的 rounds（正确）：**

```
正方一辩陈词
反方一辩陈词
主持人提问（第一轮）     ← 包含正方回应 + 反方回应
正方一辩陈词（第二轮）
反方一辩陈词（第二轮）
主持人提问（第二轮）     ← 包含反方回应 + 正方回应
反方结辩陈词
正方结辩陈词
```

与文字稿中实际标注的环节一一对应。主持人提问 round 的 content_summary 正确包含了辩手的回答内容，claims 通过新增的 `side` 字段正确归属到对应辩方。

### 跨版本对比

| 评估维度 | v0.4 | v0.4.1 | v0.4.2 |
|------|------|------|------|
| **Round 结构** | 立论/质询/自由辩论/结辩（硬套竞技辩论） | 同左 | ✅ 正方陈词/反方陈词/主持人提问/结辩（匹配哲理辩） |
| **质询回应** | 🔴 整段丢失 | ✅ 回收 | ✅ 回收 + 正确归属 |

### 仍然存在的问题

🟡 **摘要仍然偏泛化。** one_sentence_summary 和 Top 3 没有捕获这场辩论的独特叙事特征（「黑魔法/白魔法/魔法师」的三段式框架）。这个问题在 v0.4 → v0.4.2 的三个版本中一直存在，说明是 prompt 中 quick_view 指令的问题，而非 cleaning 或 round 结构的问题。

🟡 **responded 全部为 false。** 虽然驳论链（rebuts_claim_id）从 2 条增加到 7 条，但所有 claim 的 responded 仍为 false。对于哲理辩这种长陈词格式（6-8 分钟），反驳和回应之间隔了很长的距离，AI 难以判断回应关系。这不是当前版本的优先修复项——从 v0.4 的 0% responded 误标到 v0.4.1 的 22% 正确追踪，再到 v0.4.2 的回归保守（宁可不标，不比错标），这本身是合理的。

🟢 **主持人提问 round 将两个不同的问题合并在一个 round 中。** 第一轮主持人提问实际包含「问正方（钟表）」和「问反方（黑魔法心态）」两个独立问答，第二轮也类似。虽然 content_summary 覆盖了两边的回答，但拆分为独立 round 会更精确。

## 三次迭代的 PM 叙事

> 从 v0.4 到 v0.4.2 的三次迭代展示了一个典型的 AI 产品优化路径：
>
> **v0.4（3.4）**：全链路跑通，但输入质量（原始 AI 字幕无标点/无发言人）是瓶颈。质询回应丢失，responded 全 false。
>
> **v0.4.1（3.8，+0.4）**：在 compress 前插入 cleaning 步骤。这是 ROI 最高的一次改动——用一次 LLM 调用换来了完整度和逻辑保真各 +1 分。
>
> **v0.4.2（3.8，持平）**：修复 round 结构硬编码问题。综合分没变，但这是一个「正确性」修复——把错误的「自由辩论」从输出中删掉，比得分高低更重要。如果产品输出中出现了不存在的辩论环节，用户信任会瞬间归零。
>
> 三个版本的递进逻辑是：**先跑通 → 再提纯（cleaning）→ 再纠错（赛制自适应）**。每一步解决的是不同层次的问题，而非对同一个 prompt 反复调参。
>
> 当前剩余瓶颈是摘要质量（3 分），但这个维度的提升需要改动 quick_view 的 prompt 指令（要求捕获辩论叙事特征），而非 pipeline 架构。
