## 版本变更说明

v0.2 → v0.3 做了三个改动：

| 改动 | 类型 | 针对问题 |
|------|------|---------|
| 辩论文字稿按辩论环节重新分段，去除环节内多余空行 | 数据预处理 | `\n\n` 碎片化导致 chunk 内发言次数过多 |
| `max_tokens` 4096 → 8192 | 参数调整 | LLM 输出超过限制被截断 |
| JSON 解析失败时 raise 报错 + 自动重试一次 | 容错机制 | 静默丢失数据无法发现 |

预期效果：辩论三、四从"不可用"恢复到辩论二的水平。

## 修复效果判断

### 显著改善

1. **回合覆盖率从碎片恢复到完整。** 辩论三从 1 个环节 → 14 个环节，辩论四从 1 个环节 → 27 个环节。根因（`\n\n` 碎片化 + max_tokens 截断）被修复。

2. **辩论二 quick_view Top3 从空到有。** 证明 merge prompt 在 4-chunk 场景下可以生成合格的 Top3。

### 仍未解决

1. **quick_view 的 `hottest_clash` 和 `key_unresolved` 在所有辩论中系统性为空。** 这是下一步要修的最高优先级问题——merge prompt 需要硬约束。

2. **辩论三、四、五的 quick_view Top3 仍为空。** 辩论二（4 chunks）Top3 正常但辩论三/四（2 chunks）为空——chunk 数量可能是变量。也可能是 merge 调用的 `max_tokens=2048` 不够。

3. **辩论五退化。** 预处理增加了文件大小，导致 single-pass → chunked，摘要质量反降。解决方案：提高 single-pass 阈值到 ~50000 字符，或修复 merge prompt 后此问题自然消失。

4. **评委点评仍混入辩论回合。** prompt 未区分辩手和评委。

### 新发现

| 发现 | 证据 | 影响 |
|------|------|------|
| chunked 路径 merge 阶段的 quick_view 质量与 chunk 数量正相关 | 辩论二（4 chunks）Top3 正常，辩论三/四（2 chunks）Top3 为空 | merge prompt 可能需要更丰富的输入数据 |
| 预处理增加文件大小可能导致 single-pass→chunked 退化 | 辩论五 40KB→47KB | 应考虑提高 chunk 阈值或确保 merge 质量 |
| 自由辩论的预处理效果因稿而异 | 辩论三 自由辩论→2 回合（好），辩论四→15 回合（碎） | 需要统一的自由辩论处理规则 |

## 迭代叙事

> **v0.2 评估发现了两个 Critical 问题**：辩论三（买卖同罪）和辩论四（网暴）的输出覆盖率不到 10%——立论、质询、结辩全部丢失，只剩碎片化的自由辩论。
>
> **诊断过程**：排查了 chunk merge 逻辑、LLM 非确定性、max_tokens 设置，最终定位到根因——转录文字稿中自由辩论的每句反驳之间都有空行（`\n\n`），导致 chunk 内塞入 32 次发言而非预期的 5-8 次。prompt 要求为每次发言生成一个 round JSON → 输出量超限 → 截断 → 解析失败被静默丢弃。
>
> **修复**：三点联动——① 按辩论环节预处理文字稿，去除环节内多余空行；② max_tokens 从 4096 提升到 8192 作为安全网；③ JSON 解析失败改为 raise 报错 + 自动重试，不再静默丢数据。
>
> **验证**：重跑全部辩论，辩论三从 1.6 → 3.4（+1.8），辩论四从 1.4 → 3.2（+1.8），综合平均分从 2.72 提升到 3.52。quick_view Top3 在辩论二中首次成功填充。
>
> **新发现**：辩论五因预处理后文件变大被推入 chunked 路径导致摘要质量退化，以及 quick_view 的 clash/unresolved 字段仍系统性为空——已定位到 merge prompt 缺乏硬约束，预计 10 分钟可修复。
