# 产品需求文档（PRD）

## 1. 背景与问题定义

我妹妹陈琳大一加入校辩论队，每周至少看三场辩论赛视频学技术。上周我问她其中一场反方的核心论点是什么，她想了十秒钟说不上来。她不是没认真看——看了一个小时还拖了进度条回看关键片段。但信息太多，正反方交替太快，看完脑子里是一团线。

她不是一个人。辩论爱好者看视频学辩论，投入时间不少，收获却很薄——看完留不下结构化的认知，想讨论没地方张嘴。

**现有方案的不足**：

| 现有方案 | 问题 |
|---------|------|
| 手动看视频+做笔记 | 一小时视频看完什么都没有记住，再次观看视频手写笔记至少需要两个小时。整理完笔记，三天后忘了。没有结构化留存 |
| 通用 视频转文字AI 模型（Gemini/通义听悟） | 不懂辩论结构——分不清立论和质询，不知道谁在反驳谁，输出的事流水账的文字稿而非攻防逻辑 |
| 比赛复盘文章 | 99%的比赛没人写，质量看作者，时效性差，无法针对具体内容追问 |
| AI 辩论陪练（辩论星球/WitNova） | 走"AI当对手"路线，不是"AI当解说"；无法基于真实比赛视频内容进行分析 |
| B 站评论区 / 辩论社群讨论 | 碎片化、主观性强，没有结构化的攻防梳理，找不到想要的信息 |
| NotebookLM（Google） | 支持 YouTube 链接，可针对内容问答，但不支持 B 站；不理解辩论专属结构（立论/质询/自由辩/总结陈词） |
| BibiGPT | 支持 B 站链接转文字，可做内容问答，但是通用型工具，不具备辩论场景的结构化解析能力 |

**为什么现在做**：大模型的中文理解和结构化输出能力已足够支撑辩论分析。视频字幕获取链路成熟。辩论学习场景里，AI 辅助是一个被验证的需求（辩论星球近 4 万用户），但"分析已有比赛"这个方向，无论国内还是海外都没有直接竞品。

详见 [竞品分析](./competitive-analysis.md) · [用户画像](./user-personas.md)。

## 3. 目标用户与场景

| 优先级 | 用户 | 场景 | 痛点 |
|--------|------|------|------|
| P0 首发 | 入门辩手（陈琳式） | 晚上在宿舍看 B 站辩论赛，想快速理清双方论证结构 | 投入一小时却说不清反方论点 |
| P1 后续 | 辩论社指导老师 | 每周备训练课，需快速理解一场比赛并设计讨论问题 | 完整看一场+整理笔记要两小时以上 |
| P2 后续 | 辩题相关研究者/博主 | 从辩论中收集观点素材和案例 | 从50分钟比赛里摘5个有用观点，效率太低 |

**首发选择入门辩手**：触达最容易（高校辩论队）、使用频率最高（每周2-3场）、对早期产品容忍度高。

详见 [用户画像文档](./user-personas.md)。

## 5. 用户流程

```
用户打开产品
    │
    ├─ 新用户 ──→ 点击"加载示例"体验完整流程
    │
    └─ 任何用户
          │
          ▼
    粘贴 B站视频链接
          │
          ▼
    点击"开始分析"
          │
          ▼
    等待 ~15 秒（后台：字幕获取 → cleaning → compress）
          │
          ▼
    看到"3分钟速览卡片"
    ├─ 辩题 + 双方立场
    ├─ 正方 Top 3 论点  |  反方 Top 3 论点
    ├─ 最激烈交锋点
    ├─ 关键未回应问题
    └─ 关键词
          │
          ▼
    看到 AI 欢迎语 + 建议追问方向
          │
          ▼
    用户输入问题（或点击建议问题）
          │
          ▼
    AI 检索知识库（ChromaDB），返回基于原文的回答
          │
          ▼
    用户可以继续追问（无限轮，支持跨视频对比）
          │
          ▼
    完成 → 导出报告 / 分析新的辩论
```

## 6. 技术方案

### 整体架构

```
用户粘贴 B站链接
     │
     ▼
┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│ video_fetcher │ → │ transcript   │ → │  compressor │
│ 字幕获取      │   │ cleaner 清洗  │   │ LLM 结构化   │
└─────────────┘    └──────────────┘    └──────┬──────┘
                                              │
              ┌───────────────────────────────┤
              ▼                               ▼
     ┌────────────────┐              ┌───────────────┐
     │  3分钟速览卡片   │              │  深度追问      │
     │  quick_view     │              │  多轮对话      │
     └────────────────┘              └───────┬───────┘
                                             │
                                    ┌────────┴────────┐
                                    ▼                 ▼
                            ┌───────────┐    ┌──────────────┐
                            │ ChromaDB   │    │ RAG context  │
                            │ 向量检索    │    │ 跨视频对比    │
                            └───────────┘    └──────────────┘
```

### 关键技术选型与理由

| 决策 | 选择 | 理由 |
|------|------|------|
| LLM | DeepSeek Chat API | 中文能力强、成本低、兼容 Anthropic 协议 |
| 前端 | Streamlit | 快速迭代，Python 原生，适合 AI 产品 demo |
| 向量库 | ChromaDB（本地） | 零成本、零延迟、数据不出用户机器。Pinecone 等云服务在当前阶段过度设计 |
| Embedding | MiniLM-L12-v2（384维） | 轻量、中文效果好。P1 升级到 multilingual-e5-large（1024维） |
| 长文本处理 | Chunked + Merge | 超过 15000 字符的逐字稿按 `\n\n` 分段处理，各 chunk 独立分析后 LLM 合并 |
| 字幕清洗 | LLM 预处理 | 在 compress 前用一次 LLM 调用清洗：加标点、标发言人、去噪声 |
| 知识库切分 | ~500 字/块，段间 100 字重叠 | 平衡检索精度与语义完整性 |

### 关键失败模式与应对

| 失败模式 | 触发条件 | 影响 | 应对 |
|---------|---------|------|------|
| JSON 截断 | 单 chunk 内容过密，输出超 token 限制 | 回合数据丢失 | max_tokens 4096→8192；解析失败 raise + 重试 |
| quick_view 空白 | merge prompt 无硬约束 | 摘要卡片核心字段为空 | prompt 加"禁止返回空数组/空字符串"约束 |
| 自由辩论碎片化 | `\n\n` 断点过多，LLM 逐句输出回合 | 自由辩论被拆成 10+ 个碎片 | prompt 约束"自由辩论作为整体回合输出" |
| 赛制硬编码 | 哲理辩被套入标准竞技辩模板 | 输出不存在的主持人提问/自由辩论 | round_name 根据文字稿实际标注自适应 |
| 检索偏斜 | 向量检索 top 5 全部来自同一场辩论 | 跨视频对比失效 | 按 debate_id 分散采样（P0 待修） |
| 译名/隐喻盲区 | MiniLM-384 对文学隐喻和译名变体不敏感 | 细节检索失效 | query expansion + embedding 升级（P1 待修） |

## 7. AI 质量评估

评估不是一次性活动，而是产品的核心基础能力。所有改进必须可度量。

### 评估体系

| 模块 | 维度 | 当前得分 | 迭代历史 | 报告 |
|------|------|:---:|------|------|
| 结构化分析 | 5维（准确度/完整度/逻辑/幻觉/摘要） | **4.24** | 2.72 → 3.52 → 4.24 | [eval-report-v0.3.1](eval-report-v0.3.1.md) |
| paste-URL 流水线 | 同上 | **3.8** | 3.4 → 3.8 → 3.8 | [eval-paste-url](eval-paste-url.md) |
| RAG 知识库 | 5维（命中率/完整性/跨视频/准确性/细节） | **3.4** | 初版 | [eval-knowledge-base](eval-knowledge-base.md) |

### 结构化分析评估框架

每场辩论标注 ground truth（人工提取的真实论点、反驳关系、关键交锋），AI 输出与 ground truth 对照评分。

| 维度 | 核心问题 | 错误严重度 |
|------|---------|-----------|
| 论点识别准确度 | AI 提取的论点原文里有吗？归属对吗？ | 🔴 Critical / 🟡 Major / 🟢 Minor |
| 论点覆盖完整度 | 关键论点漏了吗？quick_view 字段全吗？ | 同上 |
| 逻辑结构保真度 | 驳论关系对吗？辩论结构完整吗？ | 同上 |
| 幻觉检测 | AI 编了不存在的内容吗？ | 同上 |
| 摘要决策质量 | quick_view 抓住了本质吗？ | 同上 |

### RAG 知识库评估框架

10 个预设查询（涵盖概念搜索、细节检索、跨视频对比、译名盲区、文学隐喻），每个查询有预期命中段落。

| 维度 | 核心问题 |
|------|---------|
| 检索命中率 | 搜一个概念，能找到所有相关原文吗？ |
| 段落完整性 | ~500 字段落语义完整吗？ |
| 跨视频对比 | 问对比问题，AI 用了多场辩论的内容吗？ |
| 引用准确性 | AI 引用的原文真的在知识库里吗？ |
| 细节可检索性 | 搜具体意象/人名/事件能找到吗？ |

### 上线后指标（需部署后收集）

| 指标 | 定义 | 为什么重要 |
|------|------|-----------|
| 首次体验完成率 | 粘贴链接 → 看到速览卡片 → 至少追问一个问题 | 用户走完了核心循环 |
| 追问深度 | 每个会话的平均追问轮数 | 速览激发了思考，参与感在起效 |
| 口口相传意愿 | 用户测试后直接问"会推荐给队友吗？" | 最朴素的产品验证 |

### 反指标

| 反指标 | 为什么 |
|--------|--------|
| AI 编造论点/数据（幻觉率上升） | 优化速度不能以编造内容为代价。一次幻觉→信任清零 |
| 分析时间 >60 秒 | 超过一分钟用户流失 |
| 用户跳过 quick_view 直接追问 | quick_view 没起到摘要作用 |
| 追问只涉及单场辩论（RAG 场景下） | 知识库的跨视频价值没体现 |

## 10. 关键产品决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 输入方式 | 视频链接而非文字稿粘贴 | 目标用户看的是 B站视频，多一步就多一层流失 |
| 首发用户 | 入门辩手而非教练/研究者 | 人数最多、频率最高、对早期产品最宽容 |
| 产品形态 | Web App 而非浏览器插件或小程序 | Streamlit 快速迭代，适合验证 |
| 向量库 | ChromaDB 本地而非 Pinecone 云服务 | 当前阶段零成本、零延迟。后续可迁移 |
| 模型 | DeepSeek Chat 而非 GPT-4 | 中文效果好、成本低、API 兼容 |
| 长文本策略 | Chunked + Merge 而非全文单次调用 | 单次上下文窗口有限，chunked 可处理任意长度 |
| 竞争策略 | 分析而非陪练 | 竞品扎堆"AI当对手"方向。差异化在"AI当解说" |
| 多赛制 | 自适应而非硬编码 | 哲理辩≠竞技辩。正确性优先于代码简洁性 |

## 11. 不做的事

| 不做 | 理由 |
|------|------|
| AI 辩论陪练（跟 AI 打辩论） | 竞品扎堆方向。我的差异化在"分析"不在"对战" |
| 社区功能（评论/点赞/分享广场） | 单人工具不需要网络效应。先让一个人能用爽 |
| 音频/视频文件直接上传 | 视频链接已解决输入问题。文件上传涉及 ASR+说话人分离，工程量大但 PM 价值低 |
| 多语言支持 | 首发只做中文。英文辩论有 Stela AI 等产品 |
| 移动端 App | Streamlit 响应式，手机浏览器够用。原生 App ROI 极低 |
| 付费墙 | 作品集用，有反馈比有钱重要 |
