## P0 — Bug，功能不可用

### #3 / #10 — 预设问题点击后无 AI 回应
- **问题**：点击建议问题按钮后，消息出现在聊天记录中但没有 AI 回复
- **根因**：`st.button` 只追加消息 + rerun，但没有触发 AI 生成逻辑
- **解决**：增加 `needs_qa_response` 标志位，rerun 后检测标志并自动调用 LLM 生成回复

### #4 — 模式切换需点两下
- **问题**：`st.pills` 的 `default` 从 `session_state` 读取，形成循环依赖，第一次点击更新 state 但 UI 跟不上
- **解决**：修复 state 与 UI 的同步逻辑

### #14 — 思维导图没渲染出来
- **问题**：Mermaid 代码以纯文本代码块形式展示，没有可视化
- **根因**：Streamlit 不支持直接渲染 Mermaid
- **解决**：使用 `st.components.v1.html` 嵌入 Mermaid.js CDN，将思维导图渲染为可交互图表，调整字号和布局使其可读

## P1 — 体验明显受损

### #9 — 加载无进度条，无时间节省提示
- **问题**：分析过程中只显示 `st.spinner` 文字，用户不知道进度也不知道要等多久
- **解决**：
  - 用 `st.status()` 多步进度替代 spinner
  - 设计弹窗卡片显示原视频时长 + "预计 3 分钟以内"
  - CSS `@keyframes` 动画驱动进度条（绕过 Streamlit 同步渲染限制）
  - 分析完成后显示节省时间：`视频时长 - 3 分钟`

### #5 — 对话排版问题
- **问题**：对话无左右区分、输入框不固定到底部
- **解决**：
  - 用户消息右对齐 + 蓝色背景，AI 消息左对齐 + 灰色背景，圆角气泡
  - 知识库输入框限定在右侧列
  - 纯 CSS 方案，不改 Python 逻辑（吸取之前改崩依赖的教训）

### #1 — 深色模式字体不可见
- **问题**：标题颜色硬编码（`#888`、`#bbb`），深色背景下与底色融为一体
- **解决**：改用 `rgba()` 半透明颜色，明暗主题均可用

### #6 — 逐字稿声称"说话人分离+时间轴"但实际没有
- **问题**：feature card 描述夸大了实际能力
- **解决**：修正描述文案，与产品实际能力对齐

### #19 — 粘贴链接框太小不够醒目
- **问题**：`st.text_input` 默认尺寸，不够突出
- **解决**：调整输入框样式，增大尺寸和视觉权重

### #7 — "分析新的辩论"按钮不显眼
- **问题**：按钮在页面底部灰色分割线下方，容易被忽略
- **解决**：调整按钮位置和样式，提升可发现性

### #15 — JSON 数据没完整展示到页面
- **问题**：`unresolved_issues`、`factual_claims` 等字段只在导出报告中，页面看不到
- **解决**：在结果页新增"未解决的问题"和"事实性主张"两个展开区，完整呈现在页面上

## P2 — 锦上添花

### #2 — feature card 无法跳转
- **问题**：三个功能介绍卡片是纯 HTML `<div>`，不可交互
- **解决**：改为可点击元素

### #8 — 标题出现链接 icon 但不跳转
- **问题**：Streamlit 自动给标题加锚点 link icon，用户以为是功能入口
- **解决**：去掉误导性锚点样式

### #11 — 导出原始 JSON 不知道干嘛
- **问题**：对普通用户无用，只有开发者需要
- **解决**：删除该导出按钮，减少界面噪音

### #12 — Markdown 导出需专门 App 打开
- **问题**：导出为 .md 文件需外部 App 打开，不是即时可看的体验
- **解决**：删除"查看完整报告"展开区，将核心内容直接在页面渲染

### #16 — "加入知识库"按钮不好找
- **问题**：藏在思维导图 tab 里，跟导出按钮挤在一起
- **解决**：调整位置到更显眼的地方

### #17 — 左侧模型选择可删除
- **问题**：只有一个模型 DeepSeek，选择器没有实际意义
- **解决**：删除侧边栏模型选择器

### #18 — AI 回复旁边加"加入知识库"按钮
- **问题**：用户看到好的 AI 回复想单独存到知识库
- **决策**：暂缓。涉及 KB 新增 qa collection 和 UI 联动，需更完整的设计（用户的哪句话值得入库？去重逻辑？）
- **状态**：已关闭，留待后续迭代

### #20 — 清空/删除没有确认框
- **问题**：`st.button` 直接执行删除，无二次确认
- **解决**：增加确认对话框，防止误操作

## 延后至下一迭代的建议

另有两条建议经评估后决定留到后续版本：

### 模拟辩论功能

一位高校辩论队成员提出：希望产品不仅能分析辩论，还能让用户选择一个立场，与 AI 进行模拟辩论训练。这是一个有想象力的方向，但当前阶段的决策是延后：

- **产品阶段不匹配**：v0.x 的核心命题是"让用户快速理解一场辩论"，属于信息消费场景。模拟辩论是信息生产场景，需要全新的交互模型（多人设 Agent、回合制对话、评判反馈闭环）
- **技术复杂度高**：需要至少三个 Agent（正方、反方、裁判）协同，涉及工具调用（查询资料）、状态管理、辩论规则引擎，工程量大
- **用户验证先行**：不确定"辩论队成员"这个细分需求是否代表更广泛用户群。计划在下一轮 guerrilla testing 中专门验证这个需求强度

### 字体方案调整（Unicode 相关）

一位开发工程师建议将产品默认字体改为系统原生 emoji + Unicode 方案，以解决跨平台渲染一致性问题。经评估后延后：

- 当前 CSS 字体栈（Arial + 系统回退）在主流平台渲染正常，无已知用户投诉
- 涉及全局字体替换 + emoji 渲染兼容性测试，改动面大但收益不确定
- 优先级低于功能完整性和内容质量提升

**决策逻辑**：两条建议都有合理之处，但都偏离了当前阶段的核心目标——验证"AI 帮用户 3 分钟理解辩论论证骨架"这个价值主张。在核心价值未被充分验证之前，不分散资源做功能扩展或非必要优化。

## 优先级评判与决策逻辑

### 为什么这样分 P0 / P1 / P2

**P0 — 功能闭环断裂，必须立刻修。** 评判标准：用户的核心任务能否走完？#3（按钮无响应）和 #4（切换需点两下）直接导致"粘贴链接 → 追问问题"这个核心闭环断裂。用户点了没反应，会认为产品坏了，一次坏体验就可能流失。#14（思维导图不渲染）虽然不影响闭环，但"结构化可视化"是产品的核心差异化承诺——如果最亮点的功能实际不可用，第一印象崩塌。

**P1 — 闭环能跑通，但体验打折。** 评判标准：用户能完成任务，但过程中有阻力或困惑。#9（无进度条）让用户在分析等待中焦虑退出；#5（对话无左右区分）让多轮追问难以阅读；#1（深色模式字体消失）直接导致部分用户完全看不清内容。这些问题不会阻断任务，但会显著拉低完成率和满意度。

**P2 — 任务已完成，但不够好。** 评判标准：锦上添花，不做不影响核心体验。#2（卡片不可跳转）用户不会因此用不了产品；#11（JSON 导出）对普通用户没有意义；#17（模型选择器）只有一个选项时纯属界面噪音。P2 的共性是"做了更好，不做也没人投诉"。

### 为什么 #18 是唯一关闭的

#18（AI 回复旁加"加入知识库"）初看合理，但追问下去设计问题暴露：
- 用户的哪句话值得入库？一句话的简单追问和深度分析混在一起，价值差异巨大
- 去重怎么做？同一用户反复问相似问题，KB 里存 10 条几乎一样的 Q&A 对检索是负收益
- 检索怎么用？Q&A 对和逐字稿片段混排时，权重如何分配？

在没有清晰设计答案之前，做一个半成品按钮比不做更差——用户点了会预期某种行为，行为不一致反而制造困惑。**克制也是产品能力。**

### 优先级不是拍脑袋

实际排序还考虑了**修复成本**和**连锁风险**：
- 低成本的 CSS/文案修改（#6 #7 #8 #19）集中快速扫掉，一个下午清掉 5 条
- 中成本的交互逻辑（#5 #9 #14 #15）需要调试和验证，安排在第二波
- 涉及 Python 运行时依赖的改动（曾因 `str | None` 语法 + chromadb 兼容性问题导致 KB 挂掉），必须在改动前评估 blast radius，CSS 层能解决的绝不碰 Python

### 三波执行节奏

| 波次 | 内容 | 工时 | 策略 |
|------|------|:--:|------|
| 第一波 | P0 全部 + P2 低成本项（#6 #7 #8 #11 #17 #19 #20） | 1-2h | 快速扫掉，消除阻断性 Bug，清理界面噪音 |
| 第二波 | P1 核心体验（#5 #9 #14 #15） | 3-4h | 重点投入，提升核心路径体验 |
| 第三波 | P2 剩余（#2 #12 #13 #16） | 2-3h | 收尾打磨 |

每波结束后推到 Hugging Face Spaces 让测试者验证，确保改动符合预期再进入下一波。

## 决策原则

从这 20 条问题的处理中沉淀出的原则：

1. **先修 Bug，再做体验**：P0 功能不可用的问题优先处理，保证核心闭环能跑通
2. **CSS 层能解决的绝不碰 Python**：UI 改动限制在样式层，避免修改运行时依赖引入连锁故障
3. **删除比保留好**：无用的功能（JSON 导出、模型选择器、复制按钮）直接删除，不做"万一有人用"的假设
4. **数据要可见**：AI 产出的结构化数据不应只藏在 JSON/Word 里，应该在页面上直接呈现
5. **不确定时先不做**：#18 的设计不够清晰时，宁可不做也不做一个半成品
